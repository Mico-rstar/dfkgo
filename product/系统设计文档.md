# DfkGo — 多模态 Deepfake 检测平台后端系统设计文档

## 一、项目背景与目标

随着深度伪造（Deepfake）技术的快速发展，AI 生成的虚假图像、视频和音频已对信息安全与社会信任构成严峻挑战。本项目旨在构建一个**多模态 Deepfake 检测平台**，为用户提供便捷、高效的伪造内容鉴别服务。用户可上传图片、视频或音频文件，系统自动调用后端 AI 模型完成检测，并返回可视化的分析结果。

DfkGo 是该平台的后端服务，基于 **Go 语言 + Gin 框架**构建，承担用户认证、文件管理、任务调度与结果透传等核心职责。

### 2.2 架构说明

系统采用经典的**三层架构 + 异步任务引擎**设计，整体数据流为：**客户端 → DfkGo → ModelServer**。核心设计原则是 **DfkGo 不接触文件字节流**——所有多媒体文件均通过阿里云 OSS STS 临时凭证直传，DfkGo 仅负责签发凭证、校验文件存在性和管理元数据。

- **API 层**：基于 Gin 框架，负责请求路由、参数校验、JWT 鉴权和统一响应封装。所有接口返回统一 JSON Envelope（`code/message/data/timestamp`），业务错误通过 6 位错误码传递，HTTP 状态码统一为 200。
- **Service 层**：业务编排核心，涵盖认证、用户管理、文件管理、任务调度和 OSS 操作五大域。严格禁止持有 `*gin.Context`，通过 `errcode.Error` 向上抛出业务异常。
- **Repository 层**：纯数据访问层，基于 GORM 实现 MySQL 读写，不含任何业务判断。
- **异步任务引擎**：由 TaskQueue（基于 buffered channel 的内存队列）和 Worker Pool（默认 4 goroutine）组成，实现检测任务的异步调度与执行。

## 三、核心功能实现

### 3.1 用户认证体系

采用 **JWT（HS256）无状态认证**方案。用户通过邮箱+密码注册登录，服务端签发 24 小时有效的 JWT Token。鉴权中间件从 `Authorization: Bearer <token>` 头解析并验证令牌，将用户身份注入请求上下文供下游使用。密码存储使用 bcrypt 单向哈希，杜绝明文泄露风险。

### 3.2 文件上传与秒传机制

文件上传采用 **STS 临时凭证直传 OSS** 模式：客户端向 DfkGo 请求上传初始化，后端签发限时、限路径的 STS 凭证，客户端直接使用该凭证将文件 PUT 到 OSS，上传完成后回调 DfkGo 进行 `HeadObject` 校验确认。

系统实现了**用户维度的 MD5 秒传**：上传初始化时，后端以 `(user_id, md5)` 联合唯一索引查询，若命中则直接返回已有文件 ID，客户端跳过实际上传流程，显著提升重复文件场景下的用户体验。

### 3.3 异步检测任务流水线

检测任务采用**生产者-消费者模型**异步执行。创建任务后立即返回 `pending` 状态，任务入队后由 Worker Pool 消费：取出任务 → 校验状态 → 置为 `processing` → 按模态（image/video/audio）路由调用 ModelServer → 写入结果。客户端通过轮询接口获取最新状态。

任务状态机为：`pending → processing → completed | failed | cancelled`。系统支持**软取消**（pending 直接取消，processing 标记取消但不中断模型推理）和**启动恢复**（进程重启时，processing 态任务置 failed，pending 态重新入队），保证系统在异常场景下的数据一致性。

### 3.4 多模态结果透传

ModelServer 的检测结果（JSON）直接透传存入数据库 `result_json` 字段，DfkGo **不做结构解析和 URL 改写**。结果中的可视化图片（如 mask 图、标注图）由 ModelServer 直传至 OSS，客户端根据 `modality` 字段自行渲染对应的结果视图。这种透传设计使后端与模型输出格式解耦，新增模态时后端无需改动。

## 四、技术创新点

1. **零字节流中转架构**：后端全程不接触多媒体文件字节流，上传和结果存储均通过 OSS STS 直传完成。这使得 DfkGo 可以部署在轻量级实例上，避免了大文件传输对带宽和内存的消耗，显著降低了运维成本和水平扩展的复杂度。

2. **模态无关的透传管线**：检测结果采用 JSON 透传策略，后端不绑定任何特定模态的数据结构。当 ModelServer 新增模态或调整输出格式时，后端代码无需修改，仅需客户端适配渲染逻辑，实现了前后端与模型服务的松耦合。

3. **统一错误码体系**：设计了 6 位业务错误码（如 4011xx 认证域、4014xx 任务域），配合统一 Envelope 响应格式和 Recovery 中间件兜底，实现了从 service 异常到客户端展示的全链路标准化错误传递。

4. **轻量级内存任务队列**：基于 Go buffered channel 实现 TaskQueue，无需引入 Redis/RabbitMQ 等外部中间件，降低了部署复杂度。配合启动恢复机制和软取消语义，在单机场景下提供了可靠的异步任务处理能力。

## 五、遇到的困难与解决方法

### 5.1 大文件传输的性能瓶颈

**困难**：视频文件可达数百 MB，若经由后端中转上传，将造成严重的内存压力、带宽瓶颈和请求超时问题，且难以水平扩展。

**解决方案**：采用阿里云 OSS STS 直传方案，后端仅签发临时凭证（限时 900 秒、限定 OSS 路径），文件由客户端直传 OSS，上传完成后通过回调接口 `HeadObject` 校验。这将文件传输压力完全转移至云存储基础设施，后端实例保持轻量。

### 5.2 异步任务的状态一致性

**困难**：检测任务耗时较长（视频可达数分钟），期间服务可能重启或崩溃，导致任务状态不一致——processing 态任务永远无法完成，pending 态任务丢失。

**解决方案**：设计启动恢复机制——服务启动时扫描数据库，将所有 `processing` 态任务置为 `failed`（附加 error: "service restarted"），将 `pending` 态任务重新入队。同时，Worker 在写入结果前会检查任务是否已被取消，避免覆盖用户的取消操作。

### 5.3 模型服务调用的不确定性

**困难**：ModelServer 推理耗时不可预测，且可能返回超时、502 等异常，后端需要优雅处理而非级联崩溃。

**解决方案**：ModelClient 接口封装了超时控制（默认 300 秒）和错误捕获逻辑，任何异常均转化为任务 `failed` 状态并记录 `error_message`。Worker goroutine 通过 `recover` 机制防止 panic 扩散，保证单个任务的失败不影响整个 Worker Pool 的运行。

### 5.4 API 层的安全性设计

**困难**：对外暴露 RESTful API 需防范常见安全风险，包括越权访问、数据库主键泄露和异常信息泄漏。

**解决方案**：对外 ID 采用 `file_<uuid>`、`task_<uuid>` 格式，隐藏数据库自增主键；所有数据操作均附加 `user_id` 过滤条件，防止水平越权；Recovery 中间件捕获未处理的 panic，返回标准化错误响应而非堆栈信息。

## 六、总结

DfkGo 作为多模态 Deepfake 检测平台的后端服务，通过零字节流中转、模态无关透传、轻量级异步任务引擎等设计，在保持架构简洁的同时实现了高效可靠的检测任务调度能力。系统采用清晰的分层架构和统一的错误处理机制，具备良好的可维护性和可扩展性，为后续支持更多检测模态、引入分布式任务队列和水平扩展奠定了坚实基础。
